# Troubleshooting — every gotcha we actually hit, and the fix

*Claude consults this table too — usually you can just describe the symptom ("my render shows `(key?)`") and it applies the fix. The full list is here so you can also solve things yourself.*

| Symptom | Cause | Fix |
|---|---|---|
| **`(key?)` in the output** + a WARNING | That citekey isn't in `library.bib` — a typo, or the paper isn't in Zotero, or the export is stale | Check the spelling against Zotero's Citation Key row; if the paper is genuinely there, see the sync-gap row below |
| **New papers "missing" even though they're in Zotero** | **The sync gap:** the file Zotero exports and the file the renderer reads are two different files, so renders use a stale library | One export, ☑ Keep updated, saved **directly** to the `library.bib` your render script reads. To confirm which file is live: check its modified-time after adding a paper |
| **"n.d." where a year should be** | The Zotero record has no year | Fix the year in Zotero (pin the citekey first!) and re-render |
| **Citations broke after I cleaned up metadata** | Citekeys are derived from metadata — editing a year/author regenerated the key, orphaning every old `[@key]` | **Pin keys before fixing metadata** (right-click → Better BibTeX → Pin citation key). If it's already happened: re-pin the key back to the old value |
| **"But the citekey says `bond2015` and the paper is Bond & Fox!"** | Nothing is wrong — keys are just internal handles | The rendered citation ("Bond & Fox, 2015") comes from the entry's *fields*, never from the key. Don't chase pretty keys |
| **The .bib has every entry twice** | A stale "Keep updated" export that silently stopped firing (renamed/moved target, changed scope) | Delete the file, re-export manually (Better BibTeX format, ☑ Keep updated) to the same path |
| **Several .bib files updating themselves — not sure which is live** | "Keep updated" registrations accumulate: every export attempt with the box ticked adds another one | Zotero → Settings → Better BibTeX → **Automatic export** lists every registration. Keep **exactly one per project** — each entry's **Collection:/Library:** line reveals its scope (it should name your project's *collection*, not "My Library") — **Remove** the rest, and archive the orphaned files |
| **Output is in the wrong citation style** | Wrong or missing `style.csl` (note: with *no* style file, Pandoc defaults to Chicago author-date — not APA) | Put the style you want at `starter-kit/style.csl` (download any style free from [zotero.org/styles](https://www.zotero.org/styles)) |
| **`pandoc: command not found`** · Mac: `…/.local/bin/pandoc: No such file or directory` · Windows: `'pandoc' is not recognized as an internal or external command` | Pandoc isn't installed / not on PATH | Mac: run the `.pkg` installer from Pandoc's releases page (or `brew install pandoc` if you use Homebrew; no admin rights → unpack the zip into `~/.local/bin`, the script looks there). Windows: run the `.msi` installer, then open a **new** terminal |
| **Word references aren't "live" Zotero fields** | Correct — Pandoc writes them as formatted text | For a co-author round that needs live citation editing in Word, use the Zotero word-processor plugin for that stage (same library, same style) |
| **Non-ASCII names (é, ö, 中文) look wrong in the .bib** | BibTeX escaping quirks | Names like "Müller" or "Bannò" render fine with modern Pandoc; if problems persist, re-do the Zotero export choosing format **"Better CSL JSON"** saved as `library.json`, and in `render.sh` / `render.bat` change `library.bib` to `library.json` |
| **Stray `[[brackets]]`, a printed `%%comment%%`, `[!note]`, `==`, or an unexpected title line in the .docx** | The draft picked up Obsidian-only syntax (written in a vault?) — Obsidian hides/styles these; Pandoc prints them literally | Keep Obsidian extras in your notes, not the manuscript — the full render-tested table is in [docs/06](06-obsidian-editor.md). Or just ask Claude: *"check my draft for Obsidian-only syntax, then render"* |
| **Claude can't see Zotero (no Zotero tools in the session)** — applies to the optional Zotero-MCP connection | The session is a **cloud** session (repo picker / cloud badge), which runs on a remote computer and can never reach the Zotero on your machine; or Zotero simply isn't running | Start a **local** session — anchored to a folder on your computer, no cloud badge (check the app's default: it may be set to Cloud) — and keep Zotero open |
| **Zotero/plugin menus don't match this guide** | The tools update regularly (guide last verified July 2026, Zotero 9) | Concepts don't change: catalogue → ID card → typesetter. Current menus: [Zotero docs](https://www.zotero.org/support/) · [Better BibTeX docs](https://retorque.re/zotero-better-bibtex/) |

**Still stuck?** Re-run the known-good demo (`starter-kit/sample-draft.md`). If *that* renders, the pipeline is fine and the issue is in your library/draft; if it doesn't, the issue is the Pandoc install.
